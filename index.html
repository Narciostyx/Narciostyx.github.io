ADL与CPO

2026/01/18

ADL

  • 实参依赖查找，即查找函数调用表达式中不限定函数名称的规则集，包括对重载运算符的隐式调用
  • 如果常规的非限定查找已经找到下列任意一项，则跳过ADL：
    • 类的成员声明
    • 块作用域内的函数声明
    • 任何非函数（模板）的声明（如变量、函数对象）
  • 若触发ADL，则遍历每个实参，将其对应的关联命名空间与类加入集合：
    • 类类型：本身，直接/间接基类，外围类（嵌套），最内层封闭命名空间
      • 若为类模板，则加入模板参数所属的命名空间与类（跳过非类型模板参数）
    • 枚举类型：加入定义该枚举的最内层封闭命名空间；若枚举是类成员，则同时加入该类
    • 指针/数组：递归检查其指向的类型 T（如 T* 或 T[]）
    • 函数类型：递归检查其参数类型和返回类型
    • 成员指针：
      • 指向数据成员：检查成员类型和所属类 X
      • 指向成员函数：检查参数类型、返回类型和所属类 X
    • 重载集/取地址：检查重载集中的每个函数；若由模板标识符命名，额外检查其类型模板实参和模板模板实参（跳过非类型实参）
    • 内联命名空间：若上述集合中的命名空间是内联的，则加入其外层命名空间；反之，若某个命名空间直接包含内联命名空间，则该内联命名空间也会被加入（实现双向传播）
  • 最终与普通非限定查找结果合并，最终进行筛选（using指示被忽略，所有非函数（模板）的名称被忽略，友元函数通过ADL可见）

[!NOTE] Note

  • ADL触发条件依赖于当前是函数调用，若非函数调用则不生效
  • ADL本身即遍历当前函数参数的命名空间与类加入考虑范围，以及其相关的类与命名空间
  • ADL也是泛型代码中交换两个对象（swap(a,b)）的核心原因，即考虑用户重载，也考虑用户不重载调用std::swap(a,b)
  • ADL 可以找到完全在类或类模板中定义的友元函数（通常是重载运算符），即使它从未在命名空间级别声明过

CPO

[!NOTE] semiregular
template<class T> concept semiregular = std::copyable<T> && std::default_initializable<T>;
即可复制，也默认可构造，其行为类似内置类型（e.g. int）
operator==未定义

  • 其弥补了ADL的一些不足，比如swap函数内未使用using std::swap，而在CPO中，只需要调用std::ranges::swap(a,b)
    • CPO是遮蔽了ADL的复杂性，其内部实现可以依赖ADL
    • 如std::ranges::swap(a,b)内部可能存在using std::swap
  • 定制点对象，C++20定义，其为一个const函数对象，无状态，行为一致，SFINAE友好
    • 其类型是一个字面量（constexpr）且满足semiregular概念的类类型
    • 所有实例都相等，且可以自由复制，其副本可以交换使用
    • 无论通过左值、右值还是const/非const实例调用，效果均相同
    • 如果参数不满足要求，CPO的operator()会从重载决议中移除
  • 给定一组类型 Args...，如果std::declval<Args>()… 满足上面 cpo 的参数要求，则 ___cpo_fn_ 遵循以下要求，否则，___cpo_fn_ 的任何函数调用运算符都不参与重载决议

IO复用

2026/01/18

socket编程

  1. 服务端
    1. 首先调用socket函数创建网络协议与传输协议，如IPv4与TCP
    2. 再使用bind函数绑定IP地址与端口
    3. 调用listen函数进行监听
    4. 服务端使用accept询问内核是否有来着客户端的连接请求，若否则阻塞（默认情况下） 等待客户端连接到来
  2. 客户端
    1. 调用socket函数
    2. 使用connect函数发起连接
  3. TCP连接中，服务器内核为每个Socket维护两个队列：TCP半连接队列（完成第一次握手但没完成三次握手的连接）、TCP全连接队列（完成三次握手且状态为established的连接）
    1. 服务端的accept函数从全连接队列中拿出一个已经完成连接的socket对象返回，后续传输都使用这个socket
    2. 半连接队列的socket叫做监听socket
    3. 全连接队列的socket叫已连接socket
  4. 上述描述的对象为同步阻塞模型

多进程模型

  1. 基于传统的TCP socket编程，如果服务器要建立多个客户端，则可以使用fork函数新建子进程![[Pasted image 20260219180010.png]]
  2. 子进程只需要已连接socket（父进程accept后使用fork传递），父进程只需要监听socket
  3. 当子进程退出后，内核仍然会保留对应的一些信息，因此需要调用wait与waitpid函数
  4. 进程本身较为消耗资源，因此下介绍多线程模型

多线程模型

  1. 使用pthread_create()函数创建线程
  2. 与多进程第二点一致
  3. 为了尽可能减小系统开销，使用线程池来避免线程的频繁创建与销毁![[Pasted image 20260219180605.png]]
    • 这个队列是全局的，需要避免多线程竞争
  4. 每来一个TCP连接，如果就分配一个进程或线程，当连接数足够大时，操作系统不可能同时维护如此巨大数量的进程/线程

IO多路复用

  1. 如果为每个连接请求都分配一个进程/线程是不合适的，那么可以使用一个进程来维护多个socket，即用最少的资源管理大量的并发IO事件
  2. select/poll/epoll函数允许进程通过它们从内核中获取多个事件

select/poll

  1. select实现多路复用的方式是将已连接的socket都放在一个文件描述符集合，调用select时将这个集合拷贝到内核中，让内核通过遍历检查是否有网络事件产生
    • 此时若有事件产生，则将此socket标记为可读或可写，再把整个集合拷贝到用户态，用户态再通过遍历找到可读或可写的socket
    • 所以，对于select来讲，需要遍历与拷贝两次文件描述符集合
    • select使用固定长度的BitsMap，一般默认最大值为1024（即0-1023）
  2. poll不再使用BitsMap进行存储，取而代之是使用动态数组，以链表形式组织，突破了select的文件描述符上限

epoll

  1. epoll在内核里使用红黑树跟踪进程所有待检测的文件描述字，吧需要监控的socket通过epoll_ctr函数加入红黑树
  2. epoll使用事件驱动的机制，内核里维护了一个链表来记录就绪事件，当某个 socket 有事件发生时，通过回调函数内核会将其加入到这个就绪事件列表中，当用户调用 epoll_wait() 函数时，只会返回有事件发生的文件描述符的个数，不需要像 select/poll 那样轮询扫描整个 socket 集合，大大提高了检测的效率

[!INFO] 注意
epoll本身仍需要将数据从内核拷贝到用户空间

  1. epoll支持两种事件触发模式，分别是边缘触发（edge-triggered，ET）和水平触发（level-triggered，LT）
    1. 处于ET的情况下，当被监控的socket上有可读事件发生时，服务器端只会从epoll_wait中苏醒一次，即使进程没有调用 read 函数从内核读取数据，也依然只苏醒一次，因此我们程序要保证一次性将内核缓冲区的数据读取完
    2. 处于LT的情况下，当被监控的 socket 上有可读事件发生时，服务器端不断地从 epoll_wait 中苏醒，直到内核缓冲区数据被 read 函数读完才结束，目的是告诉我们有数据需要读取
    3. ET一般与非阻塞IO搭配使用

Reactor与Proactor模型

2026/01/18

网络通信模型

Reactor

  1. 被动过程，当接收到来着客户端事件后，根据事件类型调用对应代码处理，因此也叫做Dispatch模式
  2. 同步非阻塞网络模型
  3. 底层是IO多路复用与线程池
  4. 三事件
    1. 客户端向服务器发送连接请求（连接事件）
    2. 客户端向服务器发送写（读）请求，服务器从请求内读取内容再写入（服务器中读取）（读事件、写事件），处理完成后返回值
    3. 连接由acceptor处理
    4. 连接后会创建handler，负责处理读写事件并响应
    5. reactor负责监听与分配事件

三线程模型

单Reactor单线程模型

  1. 连接请求、业务处理、响应请求都在一个线程中
  2. 工作流程
    1. reactor监听请求后，通过dispatch分发给acceptor或handler
    2. 如果是来着客户端的连接请求，则分发给acceptor并创建一个对应的handler
    3. 如果是来着客户端的读写请求，则reactor调用对应的handler处理
    4. handler通过read->execute->send执行流程
      ![[Pasted image 20260219160841.png]]

单Reactor多线程模型

  1. 鉴于单 Reactor 单线程模式无法充分利用和发挥多核 CPU 的性能，于是就诞生了单 Reactor 多线程模型
  2. 工作流程
    1. 与前者类似，但是handler不再进行业务处理（execute），仅响应事件
    2. handler通过read读取到数据后会发给processor进行业务处理
    3. processor会在独立的子线程（线程池） 内完成真正的业务处理，再返回响应给主线程的handler
      ![[Pasted image 20260219160817.png]]
  3. 尽管采用了多线程业务处理流程，但是由于是单reactor，瞬间高发的时候会造成reacotr的性能瓶颈

多Reactor多线程模式

  1. 工作原理
    1. 父线程中mainReactor监听连接建立事件，通过Acceptor接受，将新连接分给某个子线程
    2. 子线程的subReactor分配的连接加入到连接队列中并监听，同时创建对应handler处理事件
    3. 此时handler与单Reactor单线程的handler职责一致
      ![[Pasted image 20260219161120.png]]

Proactor

  1. 当有连接、读写等IO事件时，操作系统内核在处理完事件后主动通知我们的程序代码
  2. 异步非阻塞网络模型
  3. 工作流程
    1. Proactor Initiator 负责创建 Proactor 和 Handler，并将 Proactor 和 Handler 都通过 Asynchronous Operation Processor 注册到内核；
    2. Asynchronous Operation Processor 负责处理注册请求，并完成 I/O 操作；
    3. Asynchronous Operation Processor 完成 I/O 操作后通知 Proactor；
    4. Proactor 根据不同的事件类型回调不同的 Handler 进行业务处理；
    5. Handler 完成业务处理，Handler 也可以注册新的 Handler 到内核进程；
      ![[Pasted image 20260219170157.png]]

其他

  • Reactor本质是应用程序处理IO请求，而Proactor则是操作系统内核处理IO请求
  • 就绪事件与完成事件
    1. Reactor使用就绪事件，即可以立即执行而不会阻塞，即操作系统会告诉现在进行IO操作不会阻塞，读写仍由应用程序完成
    2. Proactor则是使用完成事件，即IO操作在后台完成（数据已经读入缓冲区或已从缓冲区写出），程序只负责提交请求与处理结果
    3. 因此可以将Reactor理解成：事件到达后由操作系统通知应用程序，应用程序进行处理；而Proactor则是：事件到达后操作系统来处理，处理完后通知应用程序（处理指IO操作）
  • 三种IO方式
    1. 阻塞IO指程序使用read()询问内核，若未准备好则一直等待，阻塞程序运行
    2. 非阻塞IO指程序轮询使用read()询问内核，若内核返回未准备好则立即返回，进行下面操作，过一段时间后继续询问内核
    3. 异步IO则是发起aio_read后，立即返回，执行其他操作，当内核准备好数据后会通知程序

boost.asio

2026/01/18

基本结构

  1. 至少拥有一个IO执行上下文例如 boost::asio::io_context 对象、boost::asio::thread_pool 对象或 boost::asio::system_context。这个 I/O 执行上下文代表了 程序与 操作系统的 I/O 服务的连接，如boost::asio::io_context io_context
  2. 执行IO操作至少需要一个IO对象，如TCP套接字boost::asio::ip::tcp::socket socket(io_context)
    • IO操作

同步连接

1. 程序通过调用 I/O 对象来发起连接操作：

socket.connect(server_endpoint);

2. I/O 对象将请求转发给 I/O 执行上下文。

3. I/O 执行上下文调用 操作系统来执行连接操作。

4. 操作系统将操作结果返回给 I/O 执行上下文。

5. I/O 执行上下文将操作产生的任何错误转换为 boost::system::error_code 类型的对象。可以对 error_code 进行比较，也可以将其作为布尔值进行测试（其中 false 结果表示没有发生错误）。然后将结果转发回 I/O 对象。

  1. 如果操作失败，I/O 对象会抛出 boost::system::system_error 类型的异常。如果发起操作的代码被写成这样：

那么 error_code 变量 ec 将被设置为操作的结果，并且不会抛出异常。

异步操作

1. 您的程序通过调用 I/O 对象来发起连接操作
socket.async_connect(server_endpoint, your_completion_handler);

其中 your_completion_handler 是一个函数或函数对象，其签名如下：
void your_completion_handler(const boost::system::error_code& ec);
所需的确切签名取决于正在执行的异步操作。参考文档会指明每个操作的适当形式。

2. I/O 对象将请求转发给 I/O 执行上下文。

3. I/O 执行上下文向 操作系统发出信号，指示其应启动异步连接。
时间流逝。（在同步情况下，此等待完全包含在连接操作的持续时间内。）
4. 操作系统通过将结果放在队列中，准备由 I/O 执行上下文提取，来指示连接操作已完成。
当使用 io_context 作为 I/O 执行上下文时，您的程序必须调用 io_context::run()（或类似的 io_context 成员函数），以便检索结果。io_context::run() 的调用在有未完成的异步操作时会阻塞，因此您通常会在启动第一个异步操作后立即调用它。
6. 在调用 io_context::run() 期间，I/O 执行上下文会从队列中取出操作的结果，将其转换为 error_code，然后将其传递给 您的完成处理程序。

[!NOTE] Note
IO对象即实际执行请求的载体，其依赖于IO上下文，由其转发请求给操作系统
boost.asio报错依赖于boost.system的boost::system::system_error，因此使用前者必须把后者也链接上

socket：处理网络通信，可以建立连接，发送与接收数据，方法：async_connect，async_read，async_write，close，read_some，receive，send，wait，wait_some等
acceptor：监听端口，接受来着客户端的连接请求，方法：async_accept，open，close等
resolver：解析主机名，支持同步与异步解析，方法：async_resolve，resolve等
endpoint：表示一个网络端点，包括IP地址与端口号，方法：address，port等

  2. UDP则是在命名空间ip::udp中，其他不变，同时其不需要建立连接，因此使用ip::udp::socket::receive_from与ip::udp::socket::send_to

[!NOTE] Note
客户端流程如下：

  1. 创建io_context对象
  2. 创建tcp::resolver对象，调用tcp::resolver::resolve方法解析服务器名称与端口列表
  3. 创建tcp::socket对象，调用boost::asio::connect方法遍历上面获取的列表尝试连接
  4. 建立连接后可以使用socket的read或write方法

服务端流程如下：

  1. 创建io_context对象
  2. 创建tcp::acceptor对象监听端口
  3. 创建客户端套接字tcp::socket，调用tcp::acceptor::accept获取与客户端的连接套接字并赋值给socket
  4. 使用write或read方法

异步方法一般加上前缀async_以及传入回调函数（通常使用std::bind绑定）
使用错误处理，若为boost::system::system_error::eof，即关闭连接

getopt以及getopt_long

2026/01/18

getopt

  1. getopt与getopt_long均为解析命令行参数的C库函数，前者处理短选项，后者还支持长选项，在头文件getopt.h下
  2. 两个函数在解析过程中会设置以下外部变量：
变量名 描述
optarg 如果当前选项需要参数，则该参数的值会被存储在这个变量中
optind 指向下一个要处理的命令行参数的索引
opterr 控制是否打印错误信息
optopt 如果发生错误，则该选项字符会被存储在这个变量中
3. getopt函数原型int getopt(int argc, char * const argv[], const char *optstring);，其中optstring是包含所有短选项的字符串，每个选项字符后可以跟一个冒号 :，表示该选项需要一个参数；如果跟两个冒号 ::，表示该选项的参数是可选的
4. getopt_long函数原型int getopt_long(int argc, char * const argv[], const char *optstring,const struct option *longopts, int *longindex);

[!NOTE] Option结构体
通常用来定义长命名行选项，通常与getopt_long函数共同使用，结构体包括以下成员：
name，has_arg，flag，val
p.s.:
flag: 如果不是 NULL，则 getopt_long 会将该选项的值（val 成员的值）存储在这里所指向的变量中。如果 flag 是 NULL，则 getopt_long 会返回 val 成员的值
e.g.:
{"force",no_argument,&force,1}，代表长选项名为force，不需要传入参数，val=1，同时getopt_long返回0，且force被置为1
{"time",required_argument,NULL,'t'}，代表长选项名为time，短名称为t（flag为NULL且val=’t’），需要传入参数，同时getopt_long返回val(‘t’)

  1. getopt跟getopt_long的处理流程为：
    1. 从argv[1]开始读取数据
    2. 根据optstring识别选项，查看是否需要参数，若是，则optarg会被传入后面跟着的参数，若否，则下一步
    3. optind更新索引
    4. 返回当前匹配的选项字符，若所有字符都处理完毕，则返回-1（长选项返回遵循上面option结构体内的规则）

[!NOTE] Note

  1. 对于getopt_long函数来讲，最后一个参数一般不会使用，仅其内部当下标使用
  2. 使用getopt_long函数前必须初始化option结构体数组，option结构体数组最后一个成员必须全为NULL（getopt_long函数本身不知道option结构体结束的位置，因此需要全NULL）
  3. 有一个只处理长选项的函数getopt_long_only，一般较少使用
