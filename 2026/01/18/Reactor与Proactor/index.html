Reactor与Proactor模型

Reactor 与 Proactor 两种网络并发模式的解析与对比

网络通信模型

Reactor

  1. 被动过程，当接收到来着客户端事件后，根据事件类型调用对应代码处理，因此也叫做Dispatch模式
  2. 同步非阻塞网络模型
  3. 底层是IO多路复用与线程池
  4. 三事件
    1. 客户端向服务器发送连接请求（连接事件）
    2. 客户端向服务器发送写（读）请求，服务器从请求内读取内容再写入（服务器中读取）（读事件、写事件），处理完成后返回值
    3. 连接由acceptor处理
    4. 连接后会创建handler，负责处理读写事件并响应
    5. reactor负责监听与分配事件

三线程模型

单Reactor单线程模型

  1. 连接请求、业务处理、响应请求都在一个线程中
  2. 工作流程
    1. reactor监听请求后，通过dispatch分发给acceptor或handler
    2. 如果是来着客户端的连接请求，则分发给acceptor并创建一个对应的handler
    3. 如果是来着客户端的读写请求，则reactor调用对应的handler处理
    4. handler通过read->execute->send执行流程

单Reactor多线程模型

  1. 鉴于单 Reactor 单线程模式无法充分利用和发挥多核 CPU 的性能，于是就诞生了单 Reactor 多线程模型
  2. 工作流程
    1. 与前者类似，但是handler不再进行业务处理（execute），仅响应事件
    2. handler通过read读取到数据后会发给processor进行业务处理
    3. processor会在独立的子线程（线程池） 内完成真正的业务处理，再返回响应给主线程的handler
  3. 尽管采用了多线程业务处理流程，但是由于是单reactor，瞬间高发的时候会造成reacotr的性能瓶颈

多Reactor多线程模式

  1. 工作原理
    1. 父线程中mainReactor监听连接建立事件，通过Acceptor接受，将新连接分给某个子线程
    2. 子线程的subReactor分配的连接加入到连接队列中并监听，同时创建对应handler处理事件
    3. 此时handler与单Reactor单线程的handler职责一致

Proactor

  1. 当有连接、读写等IO事件时，操作系统内核在处理完事件后主动通知我们的程序代码
  2. 异步非阻塞网络模型
  3. 工作流程
    1. Proactor Initiator 负责创建 Proactor 和 Handler，并将 Proactor 和 Handler 都通过 Asynchronous Operation Processor 注册到内核；
    2. Asynchronous Operation Processor 负责处理注册请求，并完成 I/O 操作；
    3. Asynchronous Operation Processor 完成 I/O 操作后通知 Proactor；
    4. Proactor 根据不同的事件类型回调不同的 Handler 进行业务处理；
    5. Handler 完成业务处理，Handler 也可以注册新的 Handler 到内核进程；

其他

  • Reactor本质是应用程序处理IO请求，而Proactor则是操作系统内核处理IO请求
  • 就绪事件与完成事件
    1. Reactor使用就绪事件，即可以立即执行而不会阻塞，即操作系统会告诉现在进行IO操作不会阻塞，读写仍由应用程序完成
    2. Proactor则是使用完成事件，即IO操作在后台完成（数据已经读入缓冲区或已从缓冲区写出），程序只负责提交请求与处理结果
    3. 因此可以将Reactor理解成：事件到达后由操作系统通知应用程序，应用程序进行处理；而Proactor则是：事件到达后操作系统来处理，处理完后通知应用程序（处理指IO操作）
  • 三种IO方式
    1. 阻塞IO指程序使用read()询问内核，若未准备好则一直等待，阻塞程序运行
    2. 非阻塞IO指程序轮询使用read()询问内核，若内核返回未准备好则立即返回，进行下面操作，过一段时间后继续询问内核
    3. 异步IO则是发起aio_read后，立即返回，执行其他操作，当内核准备好数据后会通知程序
